token_miser — Benchmark Results

End-to-end benchmark results for Claude Code configuration packages. Each run is a real agent session on a real coding task — no mocks, no cherry-picking. All experiment data on this page was collected by Rubin Johnson sole experimenter on Claude Sonnet 4.6 via AWS Bedrock.

Results

15 tasks across feature, bugfix, refactor, test-writing, and sequential categories. Mix of fixture repos (tiny-cli, tiny-api) and real open-source repos (Flask, httpx). Single run per package — treat pass rates as directional, not statistically final.

8 tasks using self-contained fixture repos only. Faster to run, no network required. Good for quick iteration.

How to read this

Pass rate is the fraction of success criteria met across all tasks. Each task has 1–3 automated criteria (file exists, command exits zero, output contains a string). This is the primary quality signal.

Tokens is the sum of input + output tokens across all tasks in the suite. Lower is cheaper per run. It does not include cache hits — cached tokens cost ~10x less and are tracked separately.

Cost is the actual USD charged, including cache reads at Bedrock rates. Costs vary run-to-run due to cache state at time of execution.

Baseline is vanilla Claude Code with no CLAUDE.md — the agent's default behavior. Every other package is compared against this.

Single-run data has high variance. A package that fails one hard task will look worse than one that happened to get an easier ordering. Interpret trends, not individual numbers.

Submitting a package for benchmarking

All data on this page was collected by me on a consistent hardware + model setup. To keep results comparable, I run all experiments myself — I don't accept self-reported numbers.

Caylent colleagues may get access to run experiments directly; reach out if you're interested.

To submit a package for benchmarking:

  1. Open a GitHub issue titled benchmark request: <package-name>
  2. Include your package files (CLAUDE.md / AGENTS.md, optional settings.json)
  3. Describe the strategy in 2–3 sentences: what behavior it changes and why
  4. Include enough reproduction details that I can run the same experiment: model, suite, any env requirements
  5. If you have your own numbers, include them with your setup details — I'll run my own and compare

I'll add packages that test a genuinely new strategy dimension and post results when I have them. I reserve the right to decline packages that duplicate existing ones or that I can't run cleanly.

If your package is in the loadout-packages registry already, just link to it in the issue — no need to paste the files.
